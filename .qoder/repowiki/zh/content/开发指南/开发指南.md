## 核心组件
- 应用入口与 Provider 编排：根组件仅负责挂载 BrowserRouter、全局 Provider 链与路由，不包含业务逻辑。
- Bootstrap 自动装配：利用 Vite 的 `import.meta.glob` 构建时扫描所有域清单，零注册发现机制。
- 数据上下文 DataProvider：按 domain 动态装配 MockRepository 或 HttpRepository，统一对外暴露 Repository 接口。
- Schema 引擎 types：定义筛选、列、行操作、批量操作、工具栏、弹窗、Tab、表单等完整页面描述类型。
- 通用列表页 SchemaPage：根据 PageSchema 自动渲染筛选栏、Tab、表格、分页、操作列与弹窗，将"写页面"降维为"写配置"。
- 渲染器注册表：集中管理列渲染器，支持字符串引用与函数两种形式，便于扩展与复用。
- 领域清单 manifest：声明域元信息、菜单分组、排序、路由与 Schema 绑定。
- **AI需求分析管线：从调研到Spec生成的完整自动化流程**。

#### 六个阶段的详细流程

**阶段1：调研准备**
- **准入条件**：确定要分析的功能模块
- **操作**：Clone项目仓库，阅读pipeline.md了解流程，使用user-research.md模板制定调研计划
- **出口标准**：调研计划已制定（目标、对象、方法、时间明确）

**阶段2：调研执行**
- **准入条件**：调研计划已确认
- **操作**：按计划执行调研，原始记录存入docs/research/目录，每个session单独文件
- **命名格式**：YYYYMMDD-<主题>.md
- **出口标准**：调研记录完整、使用标准模板格式

**阶段3：需求分析**
- **准入条件**：调研记录已存档
- **操作**：使用analysis.md模板进行分析，分析文档存入docs/analysis/目录
- **命名格式**：<功能模块>-analysis.md
- **出口标准**：分析文档覆盖所有维度（用户角色、场景、功能需求、非功能需求、边界条件）

**阶段4：本地评审**
- **准入条件**：分析文档完成
- **操作**：Qoder评审Agent对文档进行评审，输出评分和问题清单
- **出口标准**：评审Agent判定"通过"（加权总分 ≥ 3.5，无任何维度 ≤ 1分）

**阶段5：GitHub复审**
- **准入条件**：本地评审通过
- **操作**：创建PR，GitHub Actions自动触发Qoder复审，在PR评论中输出评审报告
- **出口标准**：GitHub复审通过，PR可合并

**阶段6：Spec生成**
- **准入条件**：分析文档PR已合并
- **操作**：GitHub Actions自动触发Spec生成，基于审核通过分析文档生成Spec
- **出口标准**：Spec文档合入docs/specs/，可作为开发依据

#### 文档模板规范

**用户调研模板（user-research.md）**
- 调研信息：主题、日期、对象、方法、所属模块
- 调研目标：明确要解决的问题和验证的假设
- 调研记录：原始记录和关键问答
- 关键发现：基于证据的发现总结
- 洞察与机会点：提炼出的需求方向

**需求分析模板（analysis.md）**
- 基本信息：功能模块、分析师、关联调研、状态
- 用户角色定义：角色、描述、核心诉求、权限范围
- 核心业务场景：触发条件、前置条件、正常流程、异常流程
- 功能需求清单：P0/P1/P2优先级分类
- 非功能需求：性能、安全、兼容性、可用性
- 数据需求：数据类型、来源、存储方式、预估量级
- 边界条件与约束：边界条件和已知约束
- 依赖关系：内部/外部/第三方依赖
- 开放问题：待确认事项

**功能规格模板（spec-template.md）**
- 基本信息：功能名称、版本、来源分析文档、状态
- 需求概述：背景、目标用户、核心价值
- 功能设计：功能描述、交互流程、页面/组件设计、数据模型、状态机
- 接口设计：API列表和关键接口详情
- 验收标准：Given-When-Then格式的测试用例
- 依赖与约束：依赖项和约束条件
- 上线计划：分阶段发布计划
- 变更记录：版本历史

#### Qoder Agent配置

**指导Agent（requirements-analyst）**
- 配置文件：`.qoder/agents/requirements-analyst.md`
- 角色：分析师的本地教练
- 能力：指导调研方向、提供方法论、实时检查质量、给出改进建议

**评审Agent（requirements-reviewer）**
- 配置文件：`.qoder/agents/requirements-reviewer.md`
- 角色：质量守门人
- 能力：按标准评分、输出评审报告、判定通过/不通过

#### 评审标准

**调研文档评审维度**：
- 调研目标(15%)、调研对象(15%)、调研方法(10%)、原始记录(20%)、关键发现(25%)、文档规范(15%)

**分析文档评审维度**：
- 用户角色(15%)、业务场景(20%)、功能需求(25%)、非功能需求(15%)、边界条件(15%)、文档规范(10%)

**评分规则**：
- 每个维度1-5分，加权平均 ≥ 3.5分通过，任何维度 ≤ 1分不通过

#### GitHub Actions工作流

**文档评审工作流（docs-review.yml）**
- 触发条件：PR涉及docs/research/或docs/analysis/目录
- 功能：自动调用Qoder评审Agent进行文档质量评估
- 输出：在PR评论中输出完整的评审报告和评分

**Spec自动生成工作流（spec-generation.yml）**
- 触发条件：docs/analysis/的PR被合并
- 功能：基于审核通过的分析文档自动生成功能规格文档
- 输出：创建新的PR将Spec文件提交到docs/specs/目录

### 新业务域开发流程（从零到一）
步骤概览：
1. 在 src/domains 下新建域目录（如 myDomain），包含 index.ts、manifest.ts、schema.ts、types.ts、repository.ts、pages 等。
2. 编写 types.ts：定义领域实体类型与枚举。
3. 编写 schema.ts：基于 PageSchema 定义筛选、列、分页、操作、Tab 等。
4. 编写 manifest.ts：声明域名称、标签、图标、菜单分组、排序、路由与 Schema 绑定；按需引入 repository.ts 以注册 mock 数据。
5. 编写 repository.ts：使用 registerMockData(domainName, data[]) 注册初始数据（可选）。
6. 在 config/domains.config.ts 中为该域配置数据源模式（'mock' 或 'http'）。
7. 如需自定义列渲染，参考"自定义渲染器"小节进行注册。
8. 运行 dev 服务，访问对应路由验证效果。

关键要点：
- 清单 manifest 是"域的身份证"，AI Agent 与路由系统据此发现域与页面。
- Schema 驱动页面，尽量保持可序列化（字符串渲染器名），减少手写组件。
- 数据源切换只需修改 domains.config.ts，无需改动 Schema 与页面。

### 自定义渲染器开发方法

## 性能考虑
- 代码分割与懒加载：
  - 对非 Schema 页面使用 component: () => import(...) 实现路由级懒加载，减少首屏体积。
- 列表渲染优化：
  - 合理设置 Table 的 rowKey、固定列与滚动区域，避免不必要的重排。
  - 对大数据量场景，优先后端分页与过滤，前端只做轻量计算。
- 缓存策略：
  - 当前仓库未实现缓存，可在 HttpRepository 层增加响应缓存（如按查询键缓存）或结合 React Query 等方案。
- 渲染器性能：
  - 避免在渲染器中进行昂贵计算，必要时使用 useMemo 或预计算字段。
- 构建与依赖：
  - 使用 Vite 开发/构建，按需引入 antd 组件与图标，减少打包体积。
- **需求文档性能**：
  - 大型调研文档建议使用分节存储，避免单个文件过大影响加载速度。
  - Spec 文档生成后进行压缩和优化，确保知识库检索效率。

## 故障排查指南
常见问题与定位思路：
- 页面空白或报错：
  - 检查 manifest 的路由 path 与 label 是否正确，确认是否绑定了 schema 或 component。
  - 确认 Schema 中的 entity 是否与 DataProvider 注册的 domain 一致。
- 数据为空或筛选无效：
  - 确认 repository.ts 是否调用了 registerMockData 注册数据。
  - 检查 domains.config.ts 中该域的模式是否为 'mock'，或 HTTP 接口是否可用。
- 列渲染异常：
  - 确认 render 字段使用的渲染器名已在 renderers/index.ts 中注册。
  - 查看控制台是否有"Renderer not found in registry"的警告。
- 导航失败：
  - 检查 navigateTo 模板中的占位符（如 :id）是否与记录 id 匹配。
- 性能问题：
  - 在浏览器 Performance 面板观察长任务，定位渲染器或数据处理瓶颈。
- **代码审查相关问题**：
  - 检查 GitHub Actions 日志，确认 CI/CD 流水线执行情况。
  - 查看 AI 代码审查报告，根据建议修复代码质量问题。

## 结论
通过"清单 + Schema + 渲染器 + 仓库 + AI需求分析"的分层设计，本项目实现了高内聚、低耦合的可扩展后台架构。新业务域的开发可以聚焦于类型、Schema 与少量交互逻辑，借助统一的渲染器与数据源抽象快速落地。

**新增的AI驱动需求分析流水线**进一步提升了项目开发的整体效率和质量：
- 标准化的需求分析流程确保需求质量
- 自动化的文档评审和Spec生成提高开发效率
- 完整的CI/CD集成实现持续交付
- 基于模板和Agent的项目架构指导为新成员提供清晰的学习路径

建议在后续迭代中逐步完善HTTP仓库、缓存与监控能力，进一步提升稳定性与性能。同时，持续优化AI Agent的配置和评审标准，提升需求分析和代码审查的智能化水平。

  - 在 types.ts 定义领域类型

  - 在 repository.ts 中注册 mock 数据（可选）
  - 在 domains.config.ts 中配置数据源模式

  - 如需自定义列展示，在 renderers/index.ts 中注册并使用
- 验证
  - 启动 dev 服务，访问路由验证功能与样式

### Git 工作流检查清单
- 分支管理
  - 从 develop 分支创建 feature/* 分支
  - 完成功能后创建 Pull Request
- 代码质量
  - 运行本地 ESLint 检查
  - 确保 TypeScript 编译通过
  - 遵循 Conventional Commits 提交规范
- 代码审查
  - 等待 AI 自动审查完成
  - 根据审查报告修复问题
  - 获得至少一位团队成员批准
- 合并与部署
  - 合并到 develop 分支
  - 触发 CI/CD 流水线
  - 验证测试环境部署成功

### 需求分析工作流检查清单
- 调研阶段
  - 制定调研计划（目标、对象、方法、时间）
  - 使用 user-research.md 模板记录调研过程
  - 保存原始记录到 docs/research/ 目录
- 分析阶段
  - 使用 analysis.md 模板进行需求分析
  - 覆盖所有必要维度（用户角色、场景、功能需求等）
  - 通过 Qoder 指导 Agent 实时检查质量
- 评审阶段
  - 本地评审通过（加权总分 ≥ 3.5）
  - GitHub 复审通过
  - 文档符合模板规范和评审标准
- Spec 生成
  - 分析文档 PR 合并后自动触发 Spec 生成
  - 确认生成的 Spec 文档质量
  - 将最终 Spec 合入 docs/specs/ 目录
